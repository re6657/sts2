# 卡牌组选择修复设计

## 问题

`NChooseABundleSelectionScreen` 中的机器人反复尝试选择第一个
`NCardBundle`，但界面没有响应。日志将每次尝试记录为成功，导致选择流程
陷入循环。

## 根因

`NCardBundle` 暴露的 Godot 信号是 `Clicked`，并提供生成的强类型方法
`EmitSignalClicked()`。当前实现发送通用的 `"pressed"` 信号；该调用不抛
异常，但也不会触发卡牌组的选择逻辑。代码因此错误地把“未抛异常”当作
“选择成功”，并跳过后续点击方式。

## 设计

1. 始终确定性地选择界面中的第一个有效卡牌组，保证多人锁步一致。
2. 首选调用 `NCardBundle.EmitSignalClicked()`，走游戏原生的强类型信号。
3. 发出信号后不立即宣称选择成功；等待后续帧观察确认按钮是否启用或界面
   是否关闭。
4. 若经过受控的重试间隔仍未发生状态变化，再调用
   `NCardBundle.Hitbox.ForceClick()` 作为兼容回退。
5. 禁止每帧同时触发多种点击方式，避免重复提交和多人状态分歧。
6. 日志分别记录“发出选择请求”“确认已启用”“回退点击”和“超时”，不再
   使用误导性的 `OK` 文案。

## 测试

- 新增回归测试，要求 `BundleDecider` 使用 `EmitSignalClicked()`。
- 新增回归测试，禁止恢复 `EmitSignal("pressed")`。
- 运行全部核心测试。
- 运行 Release 构建并确认 DLL 部署成功。

## 范围

本次只修改卡牌组选择器及其回归测试，不修改卡牌组评分、卡牌奖励选择、
Host 手动模式或其他未提交的工作区文件。
