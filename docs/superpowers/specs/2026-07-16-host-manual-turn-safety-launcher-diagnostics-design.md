# Host 全手动、Bot 回合安全与启动器诊断设计

## 目标

本次改动解决三个相互关联的问题：多人 Host 的卡牌选择仍可能被全局算法接管；Bot 偶发在回合状态尚未完成刷新时结束回合；启动器与日志难以快速定位具体窗口、回合和自动化来源。

最终行为是：Host 的所有游戏操作始终由玩家完成；Bot 只有在本地回合状态可靠稳定后才会求解或结束回合；启动器采用侧栏工作台，并能自动归类和展示关键异常。

## 已确认根因

### Host 卡牌选择被接管

`AutoSlayNode._Ready()` 当前无条件调用 `CardSelectCmd.UseSelector()`。该选择器是全局钩子，游戏原生的升级、删除、转换、附魔以及部分战斗中选牌会直接调用它，不会经过 `DispatchOverlay()` 的 `IsHostManualMode` 判断。因此覆盖层 Guard 正确也无法保证 Host 手动操作。

### Bot 疑似满能量跳过回合

现有总日志由多个游戏进程交错写入，视觉上看到的窗口、日志中的 `NetId` 和另一个 Bot 的状态容易混淆。日志也确认存在求解发生在能量为 0、抽牌尚未完全稳定的边界状态。当前门控主要依赖 `PlayerActionsDisabled`、手牌数量稳定和固定延迟，不能证明回合刷新动作已经完成。

### 诊断困难

当前文本日志没有稳定的实例标识和结构化异常类型。Host、Bot1、Bot2 的消息混排，且每帧调试输出淹没关键状态转换，无法直接回答“哪个 Bot、哪一回合、为什么结束”。

## 方案选择

采用“统一自动化策略 + 回合就绪门控 + 结构化诊断 + 侧栏启动器”方案。

不采用仅追加局部判断的方案，因为全局选择器和多个兜底路径会继续绕过零散 Guard；不在本阶段彻底重写 `AutoSlayNode`，以控制多人锁步风险。

## 架构

### AutomationPolicy

新增纯逻辑策略类，集中回答当前实例是否允许执行某类自动操作。操作类别至少包括：

- 注册全局卡牌选择器；
- 战斗出牌和结束回合；
- 卡牌升级、删除、转换、附魔；
- 事件、奖励、商店、休息点与地图决策；
- LLM 选择和超时兜底。

多人 Host 对所有交互操作一律返回拒绝；Bot 按现有自动化配置返回允许；单人模式保持当前开关语义。所有 Decider、覆盖层分发、随机兜底和选择器注册均使用同一个策略入口。

Host 启动时不调用 `CardSelectCmd.UseSelector()`。退出时仅在确实注册过选择器的情况下释放作用域。

拒绝事件写入一次性的 `[HOST-MANUAL]` 诊断，不允许按帧重复写入。

### TurnReadinessGate

新增可测试的纯状态门控器，将“可以开始求解”和“可以确认结束回合”拆成明确判断。输入快照包括：

- 本地 `NetId`、回合编号和玩家行动禁用状态；
- 当前能量、手牌数量、抽牌堆数量、弃牌堆数量；
- 可打出卡牌数量及其 ID；
- 动作队列是否空闲；
- 是否仍在抽牌或执行回合开始动作；
- 同一快照连续稳定帧数；
- 是否已经提交结束回合请求。

只有本地回合有效、动作队列空闲、抽牌完成、状态连续稳定且尚未提交结束请求时才允许求解。

能量为 0 且手中存在正费用牌时，门控不能立即确认空计划。它进入短暂重检状态，等待能量或动作队列变化。只有连续多次获得同一稳定快照，并且 `CanPlay` 也确认没有可用牌，才允许结束回合。

任何成功出牌、回合编号变化、能量变化、手牌变化或动作队列状态变化都会重置稳定计数。

### 结构化诊断

新增诊断事件模型和写入器。每个实例根据配置中的角色写入独立文本日志与 JSONL：

- `logs/host.log` 与 `logs/host.events.jsonl`；
- `logs/bot1.log` 与 `logs/bot1.events.jsonl`；
- `logs/bot2.log` 与 `logs/bot2.events.jsonl`；
- `logs/bot3.log` 与 `logs/bot3.events.jsonl`。

每条结构化事件包含：时间、会话 ID、实例角色、角色名、NetId、房间、回合、事件类型、能量、手牌、可打出牌、计划、结束回合原因、动作队列状态和附加信息。

首批异常类型固定为：

- `TURN_SKIPPED_WITH_PLAYABLE_CARDS`；
- `HOST_AUTOMATION_BLOCKED`；
- `STATE_DIVERGENCE`；
- `ACTION_QUEUE_STALLED`；
- `OVERLAY_STUCK`。

普通状态只在状态转换时记录。每帧心跳保留在详细级别，不进入默认诊断视图。

### Launcher 侧栏工作台

启动器采用方案 B。左侧固定导航为：大厅、Bot、对话、诊断、设置。右侧使用单一内容区域切换页面。

大厅页显示 Host 和各 Bot 卡片、游戏角色、AI 人格、连接状态和手动/自动状态。Host 卡片固定显示“完全手动”，不再提供会产生歧义的 Host 自动化选项。

Bot 页编辑 1–3 个 Bot 的游戏角色和人格。对话页显示按 Bot 数量分组的聊天日志入口。设置页保存种子、API 和高级参数。

诊断页采用 A 级功能：

- Host、Bot1、Bot2、Bot3 标签；
- 全部异常与五类异常筛选；
- 按时间排序的异常列表；
- 选中异常后显示前后原始日志上下文；
- “打开日志目录”和“刷新”操作；
- 错误为红色、警告为琥珀色、正常状态为绿色。

启动器不直接修改游戏状态，只读取结构化日志和原始日志。

## 数据流

启动时，配置确定实例角色。`AutomationPolicy` 决定是否注册选择器；Host 不注册，Bot 注册。日志写入器为该实例创建独立文件。

战斗中，游戏状态先转换为 `TurnSnapshot`。`TurnReadinessGate` 返回等待、可求解或可结束三种结果。只有“可求解”进入 Solver；空计划必须再次通过“可结束”判断才能提交 `EndPlayerTurnAction`。所有关键转换写入结构化事件。

Launcher 诊断页轮询 JSONL 文件的新增部分，按实例和异常类型建立索引；用户选择异常后，再从对应文本日志读取有限的前后文。

## 错误处理

日志文件暂时被其他进程占用时，写入器采用共享读写模式并进行有限重试；失败不会阻塞 Godot 主线程。结构化日志损坏时，Launcher 跳过单行并显示解析警告，不影响其他事件。

策略拒绝不抛异常。门控无法读取关键状态时返回“等待”，不得默认结束回合。只有经过稳定确认的无可用牌状态才能结束。

## 测试策略

新增独立测试项目或现有测试目标，优先测试纯逻辑组件：

- 多人 Host 对所有自动操作均返回拒绝；
- Bot 和单人模式保持预期授权；
- Host 不注册全局 `ICardSelector`；
- 能量尚未刷新时，门控返回等待；
- 满能量且存在可用牌时，门控返回可求解；
- 稳定且确实没有可用牌时，门控允许结束；
- 已提交结束请求时不会重复提交；
- 三个实例写入不同日志文件；
- Launcher 能正确分类五类异常并显示上下文。

集成验证使用 Host + 2 Bot：分别触发升级、删除、转换和战斗中选牌，确认 Host 全手动；记录至少三场战斗，确认 Bot 不会在存在可打出牌时结束；人为制造一条测试异常，确认诊断页分类和上下文展示正确。

## 非目标

本阶段不彻底拆分 `AutoSlayNode`，不修改 Solver 的卡牌评分策略，不新增远程日志服务，也不让 Launcher 主动干预运行中的游戏进程。
